\vspace*{-0.3em}
\begin{center}
  {\LARGE\bfseries\color{uoeblue}InfraQ}\\[5pt]
  {\large A Scheduling Gateway for Shared Local LLM Inference}\\[4pt]
  {\small\textcolor{uoeblue}{\faGithub}\enspace
   \href{https://github.com/bowenwan6/InfraQ}{\texttt{github.com/bowenwan6/InfraQ}}}
\end{center}
{\color{uoeblue}\rule{\linewidth}{1.0pt}}
\vspace{3pt}

\textit{InfraQ brings vLLM-style scheduling to Mac hardware, without CUDA or NVIDIA.
Four capabilities make this work: a RabbitMQ queue that decouples client acceptance from
inference so requests are never dropped under burst load;
a K-slot parallel scheduler exploiting server-side concurrency;
an exact-match Redis cache returning repeated prompts in $\sim$11\,ms;
and a Redis config channel for zero-restart strategy switching.
On repeated-prompt workloads, cache-aware dispatching achieved
\textbf{5.6$\times$ higher throughput} and \textbf{76\% lower average latency};
on unique-prompt workloads (\texttt{OLLAMA\_NUM\_PARALLEL=4}, 3~repeats),
parallel scheduling achieved \textbf{60\% higher throughput} over sequential
(0.368 vs.\ 0.230\,req/s) and \textbf{38\% lower P95} (252\,s vs.\ 409\,s),
with continuous dispatching edging static batching by eliminating batch-boundary stalls.}

\vspace{2pt}

%% ─── 1. Problem and Idea ────────────────────────────────────────────────────
\section{Problem and Idea}

Small engineering teams increasingly run LLMs on a shared Mac server instead of cloud APIs.
Without a message queue, concurrent requests pile up and latency spikes unpredictably;
even with one, a serial consumer still leaves server-side concurrency idle.
Two more problems compound this: repeated prompts are re-inferred from scratch every time,
and changing the scheduling strategy requires a container restart.

vLLM~\cite{vllm} and SGLang~\cite{sglang} solve these problems at the GPU kernel level:
continuous batching avoids batch-boundary idle time, and prefix caching reuses KV tensors
across shared prompt prefixes. Both require NVIDIA hardware and CUDA toolchains that
most small teams cannot justify.

InfraQ re-implements both techniques at the infrastructure layer, making them available
on any local LLM server (Ollama~\cite{ollama}, LM~Studio, llama.cpp), including
Mac~M1---no NVIDIA required. The four capabilities in the abstract are the result.

%% ─── 3. Architecture ────────────────────────────────────────────────────────
\section{Architecture}

The Spring Boot gateway returns HTTP~202 immediately, then writes to PostgreSQL,
caches job status in Redis, and enqueues to RabbitMQ---none of these block on inference,
so the gateway remains responsive under burst load.
The Python asyncio worker consumes from RabbitMQ and applies the active scheduling
strategy. Ollama runs natively on the host, reached via \texttt{host.docker.internal},
preserving Mac Metal GPU acceleration without container passthrough.

Redis serves four roles: browser-polling fast-path ($<$1\,ms), prompt-result cache,
atomic metric counters, and the live configuration channel (dashed in Figure~\ref{fig:arch}).
PostgreSQL stores full request histories and per-request timing for offline analysis.
On a configuration change, the worker drains in-flight tasks and restarts under
the new strategy.

With all prompts unique the cache cannot fire, isolating the scheduler.
With \texttt{OLLAMA\_NUM\_PARALLEL=4} enabling true concurrent inference,
parallel strategies reach 0.364--0.370\,req/s versus 0.230\,req/s for sequential---a
60\% throughput gain. Continuous dispatching edges static batching by 1\% throughput
and 4\% P95 (252\,s vs.\ 263\,s): it refills a slot the moment any request completes,
whereas static batching waits for the whole batch before freeing any slot.

\label{tab:expb}
\small\setlength{\tabcolsep}{6pt}
\begin{tabular}{lcrrrr}
\toprule
Strategy          & Slots & Avg latency & P95     & Throughput   & Hit rate \\
\midrule
Sequential        & 1     & 311\,s      & 584\,s  & 0.165\,req/s & 0\%  \\
Static batching   & 4     & 298\,s      & 557\,s  & 0.173\,req/s & 0\%  \\

With 71\% hitting the Redis cache, cache-aware dispatching delivers a
5.6$\times$ throughput increase (0.961 vs.\ 0.170\,req/s) and 76\% lower latency
(73\,s vs.\ 299\,s). With only 29 in 100 reaching Ollama, P95 tail latency collapses
from $\sim$565\,s to 96\,s (83\%).
A slot-count ablation (Appendix~B) shows 4 slots outperforms 8 under caching:
more concurrency is not always better.


%% ─── 5. Implementation ──────────────────────────────────────────────────────
\section{Implementation}

All four strategies are implemented, tested, and live-switchable. Deployment is a single
\texttt{docker compose up}---Ollama on the host is the only external dependency.
The web UI exposes a chat tab, live metrics, and the benchmark lab used in §4.

The caching layer is prototype-grade: InfraQ stores exact-match text results rather than
KV tensors, so semantically similar prompts still miss. A production system would replace
this with embedding-based semantic matching.

\section*{Appendix A: Experimental Setup}

All experiments ran on a MacBook Air M1 (8\,GB) using Qwen2.5:1.5b via Ollama,
100 requests per run.
Experiment~A used \texttt{OLLAMA\_NUM\_PARALLEL=4} (enabling true concurrent inference)
with 3~repeats; Experiment~B used default Ollama settings (serial inference) with
5~repeats.
Unique-prompt runs drew 100 distinct prompts (40 topics $\times$ 4 templates);
repeated-prompt runs cycled an 8-prompt hot set at 80\% frequency,
yielding a 71\% average cache hit rate.

\section*{Appendix B: Slot-Count Ablation}

Increasing slots from 4 to 8 under cache-aware dispatching actually
\textit{reduces} performance: throughput falls to 0.699\,req/s and the cache hit rate
drops to 67\%. The cache starts cold, so 8 simultaneous misses flood Ollama before any
result reaches Redis; with 4 slots, fewer cold misses compete and the cache warms faster.
Optimal slot count depends on workload repetition rate and warm-up dynamics.
